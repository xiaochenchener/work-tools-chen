0. 总览：这份代码现在应该怎么理解？

最终版代码已经从“只能直接运行的脚本”升级成了“可以被外部程序调用的单文件计算核心”。后续开发时，建议把物理模型文件当作一个稳定的核心模块，外部程序只负责配置、调用、读取结果和绘图。

2. 两种运行模式：脚本模式 vs API 模式

脚本模式

python 双层作物温室大棚全年计算.py

  • 适合手动完整运行、导出结果、画图和检查诊断。
  • 会使用全局 CONFIG 的默认设置。
  • 默认可能导出 CSV、生成图片、做实测对比、打印控制摘要。

API 模式

run_greenhouse_simulation(config)

  • 适合参数校准、论文绘图、外部程序调用、智能控制。
  • 推荐用 create_core_config() 创建配置。
  • 默认不导出、不画图、不做实测对比导出、不打印大量日志。

3. 公开 API 速查

公开接口通过 __all__ 限定。后续开发尽量只调用这些接口。

create_default_config(overrides=None)

复制脚本模式默认配置。适合保留原脚本行为的调试场景。

配置深拷贝

create_core_config(overrides=None)

创建纯计算配置。外部调用首选，默认关闭导出、绘图和大量日志。

配置API 推荐

initialize_greenhouse_model(...)

初始化模型但不完整运行。适合实时控制、单步预测、MPC。

初始化返回 sim

run_greenhouse_simulation(...)

完整运行一个时段，返回结果 DataFrame。参数校准和绘图最常用。

完整模拟DataFrame

读取当前模型状态，返回温度、湿度、控制状态等核心值。

状态读取℃

基于外部输入推进一个时间步，返回 state 和 diagnostics。

单步预测实时控制

4. 最小可运行示例

4.1 推荐 import 方式

由于模型文件名是中文，推荐使用 importlib.util.spec_from_file_location() 加载，不要直接普通 import。

python
import importlib.util
from pathlib import Path

BASE_DIR = Path(__file__).resolve().parent
MODEL_PATH = BASE_DIR / "双层作物温室大棚全年计算.py"

spec = importlib.util.spec_from_file_location("greenhouse_core", MODEL_PATH)
if spec is None or spec.loader is None:
    raise ImportError(f"无法加载模型文件: {MODEL_PATH}")

greenhouse_core = importlib.util.module_from_spec(spec)
spec.loader.exec_module(greenhouse_core)

config = greenhouse_core.create_core_config({

8. 单步预测与实时控制

单步接口适合后续做自然语言决策系统、MPC、数字孪生和实时传感器预测。它不会自动积累完整结果表，需要外部程序自己收集 state。

python
sim = greenhouse_core.initialize_greenhouse_model(config, silent=True)

step_result = greenhouse_core.run_greenhouse_step(
    sim,
    {
        "datetime": "2026-04-10 12:00:00",
        "T_out_C": 15.0,
        "I_global_W_m2": 700.0,
        "wind_m_s": 1.5,
        "RH_out": 0.60,
    },
    dt_s=60,
    silent=True,
)

state = step_result["state"]
diag = step_result["diagnostics"]
print(state["T_air_C"], state["T_crop_C"], state["RH_in"])
print(diag.keys())

9. 实测对比与误差评价

实测对比用于判断模型是否偏高或偏低，也是参数优化的目标函数来源。

python
df_results = greenhouse_core.run_greenhouse_simulation(config, silent=True)

df_meas = greenhouse_core.io_read_measured_series(
    str(BASE_DIR / "data" / "实测值.xlsx")
)

cmp_result = greenhouse_core.compare_with_measured(df_results, df_meas)
df_cmp = cmp_result["comparison"]
df_metrics = cmp_result["metrics"]

print(df_cmp.head())
print(df_metrics)
指标含义解释
MBE平均偏差正值表示模拟偏高，负值表示模拟偏低
MAE平均绝对误差越小越好
RMSE均方根误差对大误差更敏感，校准常用目标
CVRMSE归一化 RMSE用于不同量级对比
R2决定系数越接近 1 越好，但不能单独作为温室模型评价依据
n对齐样本数参与评价的时刻数量

10. 参数校准开发模板

参数校准时不要在每轮循环里重复 import 模型文件；应先 import 一次，然后在 objective 中反复调用 run_greenhouse_simulation()。

python
def objective(params):
    df_results = greenhouse_core.run_greenhouse_simulation(
        config,
        param_overrides=params,
        silent=True,
    )

    cmp_result = greenhouse_core.compare_with_measured(df_results, df_meas)
    metrics = cmp_result["metrics"]

    rmse = float(metrics.loc[metrics["metric"] == "RMSE", "value"].iloc[0])
    mbe = float(metrics.loc[metrics["metric"] == "MBE", "value"].iloc[0])

    penalty = 0.0
    if abs(mbe) > 1.0:
        penalty += 10.0 * (abs(mbe) - 1.0)

    return rmse + penalty

推荐校准顺序

  1. R_leak_1_per_h：先解决夜间漏风/降温问题。
  2. R_quilt：再调整保温被热阻。
  3. C_cover_per_area、C_quilt_per_area：调整热惯性。
  4. deep_soil_temp_C：修正低温期整体热状态。
  5. effective_solar_gain_scale：校准白天太阳得热。
  6. effective_vent_gain_scale、fan_assist_vent_rate_1_per_h：校准通风降温能力。
  7. daytime_vent_on_T_C、daytime_vent_off_T_C：最后校准控制阈值。
速度建议：多进程优化时每个 worker 先执行一次 warmup_greenhouse_core()；API 模式下保持导出、绘图、审计明细关闭。

11.2 看到 short_simulation_skipped 是否是错误？

不是。它通常表示默认 CONFIG['weather'] 指向的天气文件不存在，所以自检跳过了默认短时模拟。只要接口存在、配置检查和外部真实调用通过，API 仍然可用。

12. 推荐后续开发路线

阶段 1：稳定 API 调用
不再改接口名；保证完整模拟返回 DataFrame；保证单步接口返回 state 和 diagnostics；保证自检通过。
阶段 2：真实数据完整时段验证
先确认天气起止时间、行数、实测对齐数量，再判断物理误差。
阶段 3：夜间误差校准
优先校准漏风、保温被热阻、覆盖层热容、土壤温度。
阶段 4：白天升温校准
校准有效太阳得热、透光率、短波分配和热惯性。
阶段 5：通风与控制校准
校准风口/风机换气率、开启阈值、关闭阈值。
阶段 6：论文绘图与智能控制
输出温度时序、残差图、控制状态图、通风率图，并接入自然语言决策系统。

13. 常见问题与排查

为什么设置了 00:00 开始，但结果从 21:00 开始？

通常是天气文件中 00:00~20:00 不存在，或工作表/时间列解析后这段时间被过滤。应检查 Excel 原始时间列、工作表名和 weather_auto_year_rollover。

为什么 API 模式没有生成 CSV？

这是正常设计。create_core_config() 默认 export_results_enabled=False。需要导出时可以手动把 DataFrame 保存为 CSV，或显式打开导出。

为什么 self_check 显示 short_simulation_skipped？

说明默认天气文件路径不存在，短时运行测试被跳过。只要外部 example 能跑，且其他检查 PASS，就不是 API 错误。

参数优化时为什么很慢？

常见原因是每次循环重复 import、重复计算视角因子、导出文件、画图、打印大量日志或首次触发 numba 编译。应保持模型只 import 一次、使用缓存、调用 warmup_greenhouse_core()。

能不能直接调用内部物理函数？

不建议。内部函数未来可能继续改动，外部程序只依赖公开 API，维护成本最低。

14. 开发检查清单

每次修改核心代码后

  • 运行 python -m py_compile 双层作物温室大棚全年计算.py
  • 运行 example_call_core.py
  • 确认 import 时不会自动跑全年模拟
  • 确认 self_check_api_layer() 主要检查 PASS
  • 确认 run_greenhouse_simulation() 返回 DataFrame
  • 确认结果包含 datetime/T_air_C/T_crop_C/RH_in

每次做新研究脚本前

  • 先用 create_core_config() 创建配置
  • 不要直接修改全局 CONFIG
  • 不要循环里重复 import 模型文件
  • 先检查实际模拟时间范围和行数
  • 先用短时段测试，再跑完整月份/全年
  • 保存结果时由外部脚本自己命名，避免覆盖
最终原则：API 已经可用后，后续重点应转向“真实数据验证 → 夜间误差校准 → 白天升温校准 → 通风控制校准 → 论文图表/智能控制”，不要再频繁改接口名。
